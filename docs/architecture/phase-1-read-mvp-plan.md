# Phase 1 Read MVP Plan

## Назначение фазы

В Phase 1 у платформы впервые появляется наблюдаемая ценность:
`mcp-read-server` перестаёт быть скелетом и начинает реально отвечать
на read-only запросы против тестовой инфобазы и её выгрузки. Все
write-, intelligence- и продуктовые сценарии остаются за пределами
фазы.

## Целевой результат

Рабочий `mcp-read-server`, который против тестовой 1С (публикация
Apache + соответствующий dump конфигурации) умеет:

- вытаскивать конфигурацию и метаданные живой базы;
- читать код и искать по dump;
- валидировать и выполнять безопасный read query;
- читать журнал регистрации;
- честно различать проблемы живой базы и проблемы dump-источника.

Всё — через общий `ToolResult` envelope и `REGISTERED_TOOLS`, без
собственного транспортного хоста (MCP hosting polish остаётся за
рамками).

## Первый набор инструментов Phase 1

### A. Live-read инструменты

- **`get_configuration_info`** — базовые сведения о конфигурации: имя,
  версия, совместимость, поставщик, режим совместимости.
  Источник: HTTP-запрос к публикации тестовой базы через Apache.
  Нужен как точка входа: без него нельзя убедиться, что «вообще та
  база и та конфигурация, которую мы ждём».
- **`get_metadata_tree`** — дерево объектов метаданных с фильтрами по
  типу (Справочники, Документы, Регистры и т.д.).
  Источник: live-адаптер к публикации.
  Нужен как первый реальный способ увидеть конфигурацию глазами агента.
- **`get_metadata_object`** — карточка одного объекта метаданных
  (тип, имя, синоним, базовые свойства).
  Источник: live-адаптер.
  Нужен для прицельной навигации агента после получения дерева.
- **`get_object_structure`** — реквизиты, табличные части, формы,
  модули объекта.
  Источник: live-адаптер.
  Нужен как минимальный способ понять «из чего состоит» объект.
- **`get_form_structure`** — структура формы: реквизиты, элементы,
  привязка обработчиков.
  Источник: live-адаптер.
  Нужен для сценариев «разобраться, почему форма работает так».
- **`validate_query`** — синтаксическая валидация 1С-запроса без
  исполнения.
  Источник: live-адаптер (через безопасный валидирующий вызов на
  тестовой базе).
  Нужен как дешёвый шлюз перед `execute_read_query`.
- **`execute_read_query`** — выполнение read-only запроса на тестовой
  базе с лимитами (таймаут, размер выборки).
  Источник: live-адаптер.
  Нужен, чтобы агент мог реально получать данные, а не только
  метаданные.
- **`get_event_log`** — чтение журнала регистрации с фильтрами
  (период, уровень, пользователь, событие).
  Источник: live-адаптер.
  Нужен для сценариев диагностики и аудита поведения системы.

### B. Dump-read инструменты

- **`read_module_code_from_dump`** — чтение текста модуля по имени
  объекта/модуля из выгрузки конфигурации.
  Источник: локальная распакованная выгрузка (`examples/demo-dumps` на
  этапе MVP).
  Нужен, потому что live-база чаще всего не отдаёт код так удобно и
  быстро, как dump.
- **`search_code`** — полнотекстовый/по-паттерну поиск по коду в dump.
  Источник: dump.
  Нужен как «grep по конфигурации», базовый навигационный инструмент.
- **`search_metadata`** — поиск по именам/синонимам объектов метаданных
  в dump.
  Источник: dump.
  Нужен как дубль `search_code`, но по именам объектов и их синонимам.

### C. Health / diagnostics инструменты

- **`check_runtime_health`** — агрегированная диагностика живой базы:
  HTTP-gateway, публикация, индекс, выгрузка, таймауты.
  Источник: набор реальных health-checks (заменяют stub-уровень
  `onec-health` на рабочие проверки).
  Нужен, чтобы агент мог одним вызовом понять, «жива ли вся цепочка».
- **`diagnose_connectivity_issue`** — human-readable диагностика
  проблемы соединения с live-базой или dump-источником с отнесением
  в `problem_code` через `onec-troubleshooting`.
  Источник: агрегирует выход `check_runtime_health` и состояния
  dump-каталога.
  Нужен, чтобы ошибки live vs dump не сваливались в одну «что-то
  сломалось», а приходили с читаемым кодом проблемы.

## Порядок реализации

Порядок выбран так, чтобы на каждом шаге существовал внятный и
проверяемый инкремент, а ранние шаги разблокировали поздние.

1. **Reальный `onec-process-runner`.** Замена stub на безопасный запуск
   внешних процессов (1cv8/unpacker) с timeouts, stdout/stderr, exit
   codes. Без него нет ни live-вызовов, ни работы с dump.
2. **Окружение и конфигурация тестовой базы.** Расширение `onec-config`
   до рабочей модели окружения (base URL, публикация, каталог dump,
   тайм-ауты, режимы доступа). Это входные данные для всех инструментов.
3. **Реальные health-checks в `onec-health`.** Перевод stub-проверок
   (`check_http_gateway_available`, `check_search_index_available`,
   `check_dump_path_exists`) на фактические проверки.
4. **Базовый runtime/adapter слой для `mcp-read-server`.** Live
   HTTP-adapter к публикации + dump-adapter к каталогу выгрузки; общий
   контекст вызова (`OperationContext` + environment).
5. **Live-read инструменты первой волны.** `get_configuration_info`,
   `get_metadata_tree`, `get_metadata_object`. Каждый подключается к
   adapter слою и возвращает `ToolResult`.
6. **Dump-read инструменты.** `read_module_code_from_dump`,
   `search_code`, `search_metadata` поверх dump-adapter'а.
7. **Query path.** `validate_query` и следом `execute_read_query`
   с лимитами выборки и таймаута.
8. **Event log и структура объектов/форм.** `get_event_log`,
   `get_object_structure`, `get_form_structure` — они менее критичны
   для того, чтобы «платформа работала», поэтому идут позже.
9. **Diagnostics wrapping.** `check_runtime_health` и
   `diagnose_connectivity_issue` поверх уже готовых live- и dump-слоёв
   и реальных health-checks.

## Что не входит в Read MVP

- write-логика (`mcp-write-server`, policy enforcement, backup/apply/verify);
- metadata changes (добавление/изменение объектов конфигурации);
- intelligence analysis (impact-анализ, dependency graph, линтинг);
- product workflows (пайплайны, сценарии, демо-кейсы поверх платформы);
- полноценный MCP hosting polish (регистрация трёх серверов как
  самостоятельных MCP-процессов с фичами рантайма, auth, prod-level
  packaging).

## Критерии приёмки Phase 1

- `mcp-read-server` реально подключается к тестовой базе через
  опубликованный HTTP-gateway.
- Live tools читают конфигурацию (`get_configuration_info`) и
  метаданные (`get_metadata_tree`, `get_metadata_object`,
  `get_object_structure`, `get_form_structure`) на живой базе.
- Dump tools читают код и ищут по выгрузке
  (`read_module_code_from_dump`, `search_code`, `search_metadata`).
- Безопасный read query работает: `validate_query` отсекает
  синтаксические ошибки до исполнения, `execute_read_query`
  возвращает выборку с уважением к лимитам.
- `get_event_log` читает журнал с фильтрами по периоду, уровню и
  пользователю.
- Diagnostics различают проблемы live и dump: `check_runtime_health`
  и `diagnose_connectivity_issue` выдают корректный `problem_code`
  и человекочитаемое пояснение.
- `dev-check` остаётся зелёным на каждом шаге (не деградирует
  skeleton import/wiring).
